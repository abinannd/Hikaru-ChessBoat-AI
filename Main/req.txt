Phase 10 – Step 4: Themes & Piece Sets

Objective

Expand the application's visual customization by introducing multiple board themes and chess piece sets.

The implementation should integrate with the existing SettingsManager and settings.json.

No gameplay or AI logic should be modified.

────────────────────────────────────────────

IMPORTANT

Reuse:

• SettingsManager

• ChessPiece

• ChessBoard

• Existing rendering pipeline

Do NOT modify:

• ChessInference

• MovePrediction

• inference.py

• chess_model.py

• board_encoder.py

• move_encoder.py

• train.py

• evaluate.py

────────────────────────────────────────────

Board Themes

Expand the board theme system.

Support at least:

• Classic

• Green

• Blue

• Brown (Wood)

• Slate (Gray)

Each theme should define:

• Light square color

• Dark square color

• Selection highlight color

• Legal move highlight color

Store theme definitions centrally.

────────────────────────────────────────────

Piece Sets

Expand piece rendering.

Support:

• Unicode

• Image Assets (Current)

Design the renderer so additional image-based piece sets can be added by simply dropping assets into a new folder and registering them in a configuration structure.

Avoid hardcoded asset paths throughout the code.

────────────────────────────────────────────

Theme Manager

Create a dedicated ThemeManager (or equivalent).

Responsibilities:

• Register available themes

• Register available piece sets

• Return colors and assets

• Validate theme names

MainWindow and ChessBoard should request themes through this manager.

────────────────────────────────────────────

Live Updates

Changing either:

• Board Theme

• Piece Set

should immediately refresh the board without restarting the application.

The current game position must remain unchanged.

────────────────────────────────────────────

Asset Organization

Organize image assets in a scalable structure.

Example:

assets/

    pieces/

        default/

        alpha/

        merida/

    themes/

Document how future piece sets can be added.

────────────────────────────────────────────

Verification

Verify:

✓ Every board theme renders correctly

✓ Selection highlight adapts to the active theme

✓ Legal move highlight adapts to the active theme

✓ Unicode pieces work

✓ Image assets work

✓ Theme switching is immediate

✓ Current game state preserved

✓ Settings persist after restart

✓ Invalid theme names revert safely

────────────────────────────────────────────

Output

Create:

phase10_step4_verification.md

Include:

• Theme architecture

• Theme definitions

• Piece-set architecture

• Asset organization

• Live refresh workflow

• Verification checklist

────────────────────────────────────────────

Constraints

Do NOT:

• Change gameplay

• Modify AI

• Change move execution

• Duplicate rendering logic

Only extend the visual customization system while keeping the existing architecture clean and reusable.
